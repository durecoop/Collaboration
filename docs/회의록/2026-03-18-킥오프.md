# 킥오프 미팅 대본

## 한 줄 요약
"우리 셋이 쓸 업무 관리 사이트를 AI한테 시켜서 만들어보자"

---

## 1. 배경: 왜 직접 만드는지 (3분)

> "우리 지금 업무 관리를 카톡이나 엑셀로 하잖아.
> 누가 뭘 하고 있는지, 마감이 언제인지 한눈에 안 보이니까 불편하잖아.
> 그래서 우리 셋이 쓸 업무 관리 웹사이트를 직접 만들어보려고 해."

### "노션이나 트렐로 같은 거 쓰면 안 돼?"

> "물론 좋은 툴이야. 근데 우리가 직접 만들면 이런 점이 달라:"

| | 기존 툴 (노션, 트렐로 등) | 우리가 만드는 것 |
|---|---|---|
| 기능 | 범용적 — 쓰다 보면 우리한테 안 맞는 부분이 있음 | 우리 업무에 딱 맞게 만들 수 있음 |
| 화면 | 정해진 화면, 커스텀에 한계 | 우리가 원하는 대로 구성 |
| 비용 | 유료 기능 많음 (인원 추가, 기능 확장) | 무료 |
| 데이터 | 남의 서버에 저장 | 우리가 관리 |
| 확장 | 원하는 기능이 없으면 답이 없음 | 필요하면 바로 추가 가능 |

> "핵심은, **우리 업무 방식에 딱 맞는 도구**를 만들 수 있다는 거야.
> 노션은 모든 팀을 위한 거지만, 이건 우리 셋만을 위한 거거든.
> 예를 들어 '긴급/일보따리' 분류나, 이벤트 공유 게시판 같은 건
> 노션에서 억지로 만들어야 하지만, 우리 건 처음부터 그렇게 만들면 돼."

### "근데 만들 수 있어? 코딩 모르는데?"

> "그게 포인트야. 지금은 Claude라는 AI가 코딩을 대신 해줘.
> 우리는 '이런 기능 넣어줘'라고 말만 하면 돼.
> 실제로 두레talk 메신저도 이렇게 만들었어. 진짜 돌아가는 서비스가 나와."

## 2. 진행 방식: 애자일 — 빠르게 만들고 빠르게 고치기 (2분)

> "개발할 때 '애자일'이라는 방법이 있어. 어렵게 생각하지 말고,
> 핵심은 **짧은 주기로 빠르게 만들어보고, 써보고, 고치는 것**이야."

### 기존 방식 vs 우리 방식

> "옛날에는 기획 6개월, 개발 6개월 해서 1년 뒤에 완성품을 봤어.
> 근데 그러면 막상 나온 걸 보고 '이거 아닌데...' 하게 돼."

> "우리는 이렇게 할 거야:"

```
1주차: 각자 원하는 기능 적기 (니즈 파악)     ← 지금 여기
2주차: 기능 정리 + 화면 설계
3주차: 기본 버전 만들기 (로그인 + 할 일 등록)
4주차: 써보고 → 피드백 → 수정
  ... 이걸 반복
```

> "1~2주마다 실제로 돌아가는 걸 보면서 '이건 좋다, 이건 바꾸자' 하는 거야.
> 그래서 **지금 니즈를 적는 게 중요해.**
> 각자 원하는 게 구체적일수록 첫 버전이 더 좋아지거든."

### 왜 니즈가 중요한지

> "같은 '업무 관리'라도 사람마다 원하는 게 달라.
> 나는 전체 현황 보기가 중요한데, 누군가는 내 할 일만 깔끔하게 보고 싶을 수 있고,
> 누군가는 완료 체크가 제일 중요할 수 있잖아."

> "그래서 각자 적어주면 세 명의 니즈를 합쳐서 진짜 필요한 기능 위주로 만들 수 있어.
> 적지 않으면 내 생각으로만 만들게 되니까, 쓰다가 불편할 수 있어."

## 4. 저장/공유 방법: 커밋, 푸시, 풀 (3분)

> "우리가 같이 작업하려면 '저장하고 공유하는 방법'을 알아야 해.
> 세 단어만 기억하면 돼: **커밋, 푸시, 풀**"

### 구글 문서로 비유하면

> "구글 문서는 여러 명이 동시에 수정하면 자동으로 합쳐지잖아.
> 우리가 쓰는 방식(Git)도 비슷한데, 자동이 아니라 직접 '저장'하고 '올리는' 거야.
> 대신 누가 뭘 바꿨는지 기록이 다 남아서, 실수해도 되돌릴 수 있어."

### 세 단어만 기억하세요

| 단어 | 뜻 | 비유 |
|------|------|------|
| **커밋 (commit)** | 내 작업을 저장하기 | USB에 저장하는 것 |
| **푸시 (push)** | 저장한 걸 팀에 올리기 | 공유 폴더에 업로드하는 것 |
| **풀 (pull)** | 팀원이 올린 걸 받아오기 | 공유 폴더에서 다운로드하는 것 |

### 실제로 하는 방법

> "전부 Claude한테 말로 시키면 돼:"

```
작업 시작할 때 → "pull 해줘"         (팀원이 올린 최신 내용 받기)
작업 끝나면   → "커밋하고 푸시해줘"    (내 작업 저장하고 팀에 공유)
```

### 왜 이 순서가 중요한지

> "꼭 **pull 먼저, 작업, 푸시** 순서로 해야 해.
> 안 그러면 이런 일이 생겨:"

```
❌ 잘못된 순서:
  소영이 파일 수정 → 푸시
  슬기가 pull 안 하고 같은 파일 수정 → 푸시하면 충돌!

✅ 올바른 순서:
  소영이 파일 수정 → 푸시
  슬기가 pull (소영 작업 받아옴) → 작업 → 푸시   OK!
```

> "매번 이것만 기억하면 돼:"

```
1. "pull 해줘"              ← 시작할 때 (항상!)
2. 작업하기
3. "커밋하고 푸시해줘"        ← 끝날 때
```

> "혹시 충돌이 나면 Claude한테 '충돌 해결해줘'라고 하면 알아서 해줘.
> 걱정하지 마!"

## 5. 뭘 적어야 하는지 (2분)

> "어렵게 생각하지 말고, 대화하듯이 적으면 돼. 이런 것들:"

- **맡고 있는 업무가 뭔지** (예: 콘텐츠 제작, 고객 응대, 발주)
- **있으면 좋겠는 기능** (예: 내 할 일만 모아보기, 완료 체크)
- **지금 업무 관리할 때 불편한 점** (예: 카톡에서 묻혀서 까먹음)
- **평소 쓰는 도구** (엑셀, 노션, 카톡, 수첩 등)
- **로그인 방식** (이름 클릭? 비밀번호?)

> "참고로 내가 적은 건 이런 거야:"
> - 긴급/일반 분류해서 할 일 관리
> - 마감일 표시, 임박하면 알림
> - 누가 뭐 하는지 대시보드에서 한눈에 보기
> - 월별로 완료한 일 모아보기
> - 이벤트 공유 게시판

> "이 정도로 적으면 되고, 더 간단해도 괜찮아.
> '내 할 일만 깔끔하게 보고 싶다' 한 줄도 충분해.
> **적어줘야 반영할 수 있으니까, 꼭 적어줘!**"
